The Story A Night Of Job Hunting, Four Ways In

One resume, four doors: which jobs fit me best?

The example is unchanged from the earlier decks in this series — everyone has looked for a job or hired for one. What changed is how many ways you can now ask the question.

1

The inputs

1 synthetic resume + real job postings
  • A two-page synthetic resume — Jordan Rivera, a fictional data-platform / governance / GenAI-architecture profile. The owner's real resume stays local-only, gitignored: this repo is public.
  • Four real postings captured 2026-07-09 and carried over verbatim: Anthropic, Bain, Gusto, Temporal — plus two that fail on purpose: ADP and Ashby/Jerry.ai.
  • Ask it four ways: jobmatch analyze, POST /analyze, a Python import, or a chat message.
2

What the agent does

backend/job_matcher/ — no agent runtime, no orchestration loop
  • Loads the resume once, then spawns one end-to-end async task per job — fetch → analyze → score → report — bounded by JOB_FANOUT_CONCURRENCY.
  • The one LLM call per job reports matched/missing skills with an exact quote from the resume as evidence. It cannot invent experience or emit a number.
  • A tool (plain code) computes the 100-point score: required 40 · preferred 20 · experience 20 · domain 20.
3

What comes back

runs/<ts>/ (CLI) or the response body (API) — same typed array
  • One typed JobOutcome per job — report or fetch failure — the same array on disk or in an HTTP response.
  • A ranking.md across jobs, plus new since this repo: a JSON Resume conversion and a rendered cover letter for the top pick.
  • The honest part, carried over unchanged: two links fail — ADP and Ashby need JavaScript. Logged, marked failed, no retry, no fake analysis.
Why this example still teaches well: the problem needs no explanation and the failures are real. What's new is that the same story now has four doors in — a terminal, an HTTP client, a Python import, or a chat message — so this deck can show one architecture serving every access pattern, not one runtime serving one story.

Construction What Is A Bolt?

A Bolt is AI-DLC's sprint — hours, not weeks

Same definition as every deck in this series. Unit of work = one checkbox in tasks.md · Bolt = the checkboxes done in one sitting · Gate = human permission. This time the scope needed eleven of them.

Three rules make it a Bolt

  1. 1A complete slice, not a half-built layer. When the Bolt ends, you have something that can be tested on its own.
  2. 2It ends in a stable state. Code compiles, the new evals pass, and the owner reviews the slice before the next Bolt starts.
  3. 3Decisions get written down. Anything changed mid-Bolt becomes a **Correction:** entry in tasks.md.
Contrast with the ancestors: talent-align had no Bolts — one long chat built everything at once. Eve's job-matcher needed four. This repo needed eleven because the scope itself grew: three access surfaces, two LLM operations, four telemetry backends. Bolt discipline is what let scope grow without losing reviewability.

Construction Architecture

Two LLM calls in the whole system — everything else is plain code

By design (ADR 0001): LLM-1 is typed extraction inside the core; LLM-2 is chat-tool orchestration inside the agent service. Neither may emit a score.

pipeline · one request, N jobs
jobmatch analyze --resume R --job J1 --job J2 ... (or POST /analyze) │ mint run_id → extract_resume_text(R) (once; fails fast on scanned PDF / .doc) │ spawn ONE end-to-end async task PER JOB (bounded by JOB_FANOUT_CONCURRENCY, default 3) task(J_i): fetch(J_i) (guards; exactly one attempt) │ fail → log, emit JobFetchFailure, task ends — siblings unaffected ▼ analyze(resume, job_text) ← the only per-job LLM call score_job_fit(analysis) (deterministic) │ gather all outcomes → list[JobOutcome] the typed JSON array, every surface CLI → runs/<ts>/ + ranking.md API → same array as the response body, nothing persisted

Why fan-out changed shape from Eve

  • Eve needed an N=1 vs N>1 fork — spawning a subagent had a real cost, so one job ran direct and many jobs ran through job-analyst.
  • Asyncio tasks have no such cliff. This repo fans out identically for 1 job or 50 — there is no fork in kind, only a gather of tasks.
  • Four surfaces, one service layer: CLI (cli.py, Typer) · REST (api.py, FastAPI — /analyze, /resume/jsonresume, /health, no /runs) · embeddable core (from job_matcher import ...) · MCP + chat.
One tool-definition surface serves two AI clients: Claude Desktop mounts mcp/index.js over stdio; the owner's existing neutral chatbot drives the same three tools through mcp/agent-service/ (LLM-2) over SSE — because the REST API already exists, a thin MCP→REST passthrough would be a round trip to nowhere.

Same rubric, same fixtures, restated as pytest

Two markers, two costs

  • pytest -m "not live" — HARD, no key, no network: 120+ tests — scoring determinism, match banding, slug naming, schema conformance, observability spans. Blocks implemented → verified.
  • pytest -m live — 11 tests, real model: grounding, single/multi-job outcome counts, prompt injection, fetch guards. Runs on the OpenAI mini tier — extraction is a cheap workload, cost policy dated 2026-07-11.
  • SOFT band ranges were recalibrated once at verification against the synthetic profile — Anthropic 84–85 strong, Bain 72 good, Gusto 79–81 good/strong, Temporal 40 weak.
A correction the HARD rule forced: the Bolt 3 smoke showed the mini model paraphrasing 5/14 evidence quotes under Eve's looser "quote or close paraphrase" wording. The prompt was tightened to exact contiguous quote only — a deliberate divergence from the Eve prompt, needed to make grounding actually hold.
Construction Security Baseline

A job posting is untrusted web text inside your prompt

The same adversarial fixture as the Eve deck, plus one guard Eve never needed: the Python backend fetches URLs itself, so a hostname blocklist is load-bearing here.

Prompt injection — structurally impossible, then tested

  1. 1There is no score to inject. JobAnalysis (Pydantic) has no field with "score" in its name — enforced by a HARD, grep-backed test, not a convention.
  2. 2Data, not instructions. Job text arrives fenced and labeled as content to analyze, never as commands.
  3. 3Evidence is a trap for lies. Every matched skill needs an exact resume quote, whitespace-normalized — invented matches fail grounding automatically.
  4. 4The attack ships with the repo. tests/live/test_prompt_injection.py runs it on every live sweep.

Network + link guards

  • Exactly one fetch attempt per source, no retry — on failure: log, record fetch_status: failed with a reason, that task ends. Siblings finish normally.
  • http/https allowlist + an SSRF hostname blocklist in fetch.py — the backend fetches URLs directly (no sandbox to hide behind), so this guard is new relative to the Eve reference.
  • HTTP 200 does not mean success. The minimum-extractable-words check catches ADP and Ashby's JS shells (17 and 8 words) the same way it did in the Eve fixtures.
  • Byte caps and a size ceiling round out the same standing baseline carried over from the earlier decks in this series.
No-go rules, now enforced by pytest instead of eve eval: a retried fetch, a fake analysis for a failed source, or an LLM-authored score — any one blocks implemented → verified.
Operations Telemetry, By Env Alone

Four backends, selected by which env vars are set — never by code

Local structured JSON logs are always on. Remote backends activate the moment their own documented env vars appear — multiple can fan out at once. Instrumentation is decorator-only (AOP): core function bodies contain zero telemetry calls.

BackendEnabled byExtra deps?
OpenObserve (REST)OPENOBSERVE_URL + org/stream/user/passnone
OpenObserve (OTel)OTEL_EXPORTER_OTLP_ENDPOINT[otel]
Arize PhoenixPHOENIX_COLLECTOR_ENDPOINT[otel]
Arize AXARIZE_SPACE_ID + ARIZE_API_KEY[otel]

Why decorators, not inline calls

  • @traced / @timed attach to key methods only. A grep gate proves core modules have no trace/span calls inside function bodies — removing the decorators must leave behaviour identical.
  • Cost policy (2026-07-11): live evals and chat both run on the OpenAI mini tier (MODEL_ANALYST=openai:gpt-5.4-mini) — extraction-only workload; upgrade only if SOFT expectations miss.
  • Delivery is fire-and-forget. An unreachable backend logs one warning and never fails a run — the same graceful-degradation rule as the fetch guards, applied to observability.
Testing and monitoring meet here too: tests/test_observability.py checks the same span nesting and parent/child relationships that Operations reviews live — one instrumentation contract, two consumers.

Where it goes next

  • frontend/ (later phase): a Vite SPA modeled on privacyshield, consuming the same FastAPI contract this deck already documents — the OpenAPI spec ships with every GitHub Release.
  • Auth & rate limiting: the API binds to localhost by default today — explicitly deferred to the deployment change, not an oversight.
  • Chat, today: the owner's neutral chatbot already drives analyze_job_fit end-to-end through the agent service — verified, not just unit-tested (ADR 0001).
